# Kognitika — программа долгосрочного развития

Этот документ связывает текущий технический backlog с расширенным
нейрокогнитивным видением из исследовательского отчёта
`docs/Исследование рынка когнитивных D2C-приложений.md`.

Цель — не переписать продукт одним прыжком, а выстроить последовательность фаз,
где каждая опирается на уже работающий фундамент.

## Фаза 0 — фундамент (завершена)

В продукте уже стабилизирован базовый слой:

- boot/recovery;
- `StorageGateway` и контракты локального хранилища;
- PII-safe logging;
- split `EventBus`;
- удаление Firebase-хвоста;
- направление Brain ID-first identity;
- build badge для QA;
- GitHub Actions deploy flow с контролем drift между git и production.

Это не просто закрытый техдолг. Это предпосылка для всего, что дальше: без
надёжного storage layer нельзя строить долгосрочные траектории пользователя, а
без разделённого EventBus новые сенсорные и аналитические события начнут
конфликтовать с продуктовой логикой.

## Фаза 1 — цикл вовлечения и обратной связи (issues #46–#50)

Горизонт: 2–3 недели.

Это ближайшая и самая важная фаза, потому что она превращает разовые тесты в
систему развития.

### #46 — рекомендация следующего теста после завершения

Каждый экран завершения должен отвечать на три вопроса:

- что получилось хорошо;
- что стоит улучшить;
- что тренировать следующим шагом.

Это техническая основа ежедневной траектории.

### #47 — ежедневная траектория проверки навыков

Модель маршрута нужно сразу закладывать шире, чем список когнитивных тестов.
В будущем туда должны войти практики когнитивной безопасности и
соматико-созерцательные упражнения.

Рекомендуемый общий тип:

```ts
type DailyPracticeCategory = 'cognitive' | 'somatic' | 'safety';
```

Лучше один общий `DailyPracticeItem`, чем отдельная модель под каждый будущий
вид практики.

### #48 — кривая когнитивных способностей

Кривая должна быть проекцией над чистым логом событий Brain ID, а не
одноразовой денормализованной таблицей.

Кандидаты событий:

- `TaskAttempted`;
- `SessionFinished`;
- `StreakBroken`;
- `PracticeRecommended`;
- `PracticeCompleted`.

Такой подход готовит продукт к Event Sourcing-подобной аналитике без полного
переписывания сейчас.

### #49 — конкретная похвала после каждого теста

Похвала должна быть привязана к измеримому изменению:

- скорость реакции;
- точность;
- устойчивость внимания;
- улучшение относительно baseline;
- восстановление после ошибок.

Это не только UX-полировка. Пользователь должен чувствовать связь между
практикой и результатом.

### #50 — аудит доставки обратной связи

Нужно ясно понимать, куда уходит пользовательская обратная связь:

- сохраняется ли она в PostgreSQL;
- видна ли в `AdminPanel`;
- уходит ли в настроенный admin notification channel;
- можно ли проверить это синтетическим smoke-тестом;
- соблюдаются ли Brain ID / 152-ФЗ ограничения.

## Фаза 2 — блок когнитивной безопасности

Горизонт: 1–2 месяца после verified-закрытия Фазы 1.

Существующие компоненты уже указывают в эту сторону:

- `SituationalJudgmentTest`;
- `DialogueArchitecture`;
- soft-skill и language-oriented тренажёры.

Практический путь — расширить диалоговые сценарии так, чтобы пользователь учился
распознавать паттерны манипуляции и выбирать здоровую реакцию.

Кандидаты паттернов:

- газлайтинг;
- ложная дихотомия;
- эмоциональное давление;
- давление авторитетом;
- love bombing / изоляция;
- пропагандистский framing;
- когнитивные shortcuts, используемые дезинформацией.

Исследовательский отчёт указывает на prebunking/inoculation-подходы, игры типа
Bad News и Harmony Square, BITE model и conversational belief reframing.

Для #48 здесь появляется второе измерение: не только память, скорость и
внимание, но и метрика `cognitive immunity`.

Acceptance horizon:

- создать typed contract `SafetyScenario`;
- добавить labels для паттернов манипуляции;
- использовать нейтральный, не алармистский русский copy;
- тестировать scoring на синтетических сценариях;
- держать модуль образовательным, а не диагностическим.

## Фаза 3 — мультисенсорное расширение через браузерные каналы

Горизонт: 2–4 месяца после verified-закрытия Фазы 1.

Отчёт говорит про визуальные, аудиальные, проприоцептивные, тактильные,
обонятельные и вкусовые каналы. Для текущей web-платформы реалистичный порядок
уже:

### 1. Аудио

Web Audio API может дать:

- звуковую обратную связь на ответ;
- синхронизацию с дыханием;
- опциональные ambient/focus modes;
- короткие attention reset cues.

Это не требует нового железа.

### 2. Haptics

Через Web Vibration API и опыт `koord-haptic` можно добавить:

- вибропаттерн на правильный/неправильный ответ;
- mobile-only haptic cues;
- pacing для дыхательных упражнений;
- accessibility toggles.

### 3. Обоняние и вкус

Оставить как research horizon, а не product commitment. Это требует
специализированного железа и должно появляться только через отдельное
hardware/research-партнёрство.

## Фаза 4 — соматико-созерцательный модуль

Горизонт: 3–6 месяцев после verified-закрытия Фазы 1.

Эта фаза использует слот `DailyPracticeItem.category: 'somatic'` из #47.

Практики, которые можно сделать нативно в браузере:

- slow-paced breathing около 0.1 Hz с визуальным таймером;
- короткие reset-практики между когнитивными тестами;
- нейтральные attention practices без религиозного framing;
- опциональный расширенный созерцательный контент только по явному opt-in.

Правило продукта: без медицинских обещаний. Описывать это как инструменты
восстановления, внимания и устойчивости практики, пока отдельный legal/research
review не разрешит более сильный язык.

Acceptance horizon:

- typed contracts `SomaticPractice` и `BreathingPattern`;
- доступный breathing timer с reduced-motion поддержкой;
- ясный safety/comfort copy;
- интеграция в daily trajectory как optional/skip-safe item;
- хранить только completion/progress events, не чувствительные health data.

## Фаза 5 — Rust/WASM и portable Brain ID

Горизонт: параллельно фазам 2–4, но только когда появится ясный hot path.

Rust/WASM — не стратегия переписывания. Это граница для вычислительно тяжёлых
или integrity-sensitive частей:

- агрегация длинной истории cognitive trend;
- антифрод-анализ паттернов ответов;
- deterministic task generation;
- portable offline scoring kernels.

Portable Brain ID становится важнее, когда профиль пользователя растёт между
устройствами и модальностями. Клиент должен хранить только session token или
recoverable handle; владение identity остаётся server-side, а export/import
должен быть явным и управляемым пользователем.

## Фаза 6 — научная валидация и партнёрства

Горизонт: 6–12 месяцев.

Исследовательский отчёт опирается на 70+ источников. Это даёт Kognitika шанс
быть больше, чем ещё одним brain-training приложением, но только если data model
останется чистой.

Стратегические требования:

- только анонимизированные агрегаты;
- без raw Brain ID, email, token и private telemetry в exports;
- event log shape, пригодный для research review;
- явные consent boundaries;
- только синтетические fixtures в публичном коде и тестах;
- partner-facing документация отдельно от runtime code.

## Принцип последовательности

Не начинать поздние фазы до verified-закрытия Фазы 1.

Если добавить мультисенсорные данные до стабилизации trajectory, trend curves и
event contracts, модель данных придётся переписывать. Та же дисциплина, которая
стабилизировала boot, storage и deploy, должна управлять продуктовым
расширением:

```text
issue -> branch -> tests -> PR -> CI -> deploy -> verified status
```

Прямые production patches запрещены, кроме документированного emergency rollback.
